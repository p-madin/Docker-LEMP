# Phase 8: Persistence & C(-r)UD Controller (Source Code Draft)

This document describes how the dynamic payloads generated by the Phase 7 Block Editor are persisted to the database via the Event Sourcing architecture.

## 1. Implementation Overview

Because the Page Builder sends complex, deeply nested JSON structures representing an entire page's layout, a specialized `PageController` was introduced to parse, validate, and serialize this state into the `event_store`.

## 2. Core Enhancements

### Granular Event Dispatching
When the client "Saves" a workspace, the `PageController` does not simply wipe the database and insert new rows. Instead, it compares the incoming JSON payload against the existing state and dispatches specific, atomic events to the `EventStore`:
- `PageCreated` / `PageUpdated` / `PageDeleted`
- `ElementCreated` / `ElementUpdated` / `ElementDeleted`

### Soft Deletion
Pages support soft deletion. Instead of destroying the records from `tblPages`, the `PageDeleted` event handler simply updates the `pagDeleted` timestamp column. This ensures historical references are maintained and pages can be cleanly restored if necessary.

### Automated Bridge Management
During the event materialization phase in `worker.php`, the handlers automatically manage the `brgPageElements` table. If an `ElementUpdated` event dictates a new parent or a new index order, the bridge table's `pelOrder` is updated accordingly.

## 3. Benefits & Outcomes
- **Auditability**: Every specific change to a CMS page (e.g., updating a button's label) is recorded as a discreet event, rather than masking the change within a monolithic JSON blob.
- **Reversibility**: The architecture seamlessly integrates with the Phase 5 Undo/Redo mechanisms, allowing administrators to roll back page edits incrementally.
